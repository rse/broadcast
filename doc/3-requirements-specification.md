
Requirements Impact
===================

The documented requirements have impacts on the software solution and user experience.
This chapter should explain the impacts and results in the current solution.
We try to state the roles involved, impact on the data model.

| Req.   | Req. Name                             | Software role | Data model (Entity - Attribute)                                                       | description                                                                                                                                                                                                                                                                                   |
|--------|---------------------------------------|---------------|---------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| R1.1   | User authentication                   | Everyone      | Event - accessList; AuthorizationToken; SessionToken                                  | Users that are part of the accessList can challenge an authorization. Successful login results in a SessionToken                                                                                                                                                                              |
| R1.2   | Limit parallel access                 | Attendee      | SessionToken - user, event                                                            | A user can only have one active SessionToken per event. If he challenges a new login, existing SessionTokens will be deleted, opened connections will be closed and the user gets a notice message.                                                                                           |
| R1.3   | Attendee scaling                      | -             | -                                                                                     | The HAPROXY, mosquitto and the server of a runtime environment (DV, QA, PR) will have several instances in order to scale for multiple connected users.                                                                                                                                       |
| R1.4   | Streaming options                     | -             | Channel - channelId                                                                   | Organizing streams in different channels allow the logical structure of language and/or resolution specific streams.                                                                                                                                                                          |
| R1.5   | Attendee interaction channel          | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.5a  | Questions                             | Attendee      | Message - type                                                                        | Each question will be a new object of entity Message with type set to __Question__                                                                                                                                                                                                            |
| R1.5b  | Chat                                  | Attendee      | Message - type                                                                        | Each chat message will be a new object of entity Message with type set to __Chat__                                                                                                                                                                                                            |
| R1.5c  | Likes                                 | Attendee      | Message - liker, likes                                                                | A Message (Chat or Question) can be liked by a liker. That relation to a liking User can be created and deleted (undo like). On event finish the bare count of likers is updated to attribute likes and the liker relations are deleted.                                                      |
| R1.5d  | Replies                               | Attendee      | Message - replyTo, Event - chatReply                                                  | A Message can have another Message as reply. They will get linked together with the replyTo relation from original Message to reply Message.                                                                                                                                                  |
| R1.5e  | Name appearance                       | Attendee      | Message - senderName; Event - chatName, questionsName                                 | An Event defines its prefered name for chats and questions. When a Message is stored it gets a senderName based on the options of the event.                                                                                                                                                  |
| R1.5f  | Display options                       | Attendee      | Message - sender                                                                      | The display option is realized in the client software only. Users could toggle the events chat or see a different area with only their own messages.                                                                                                                                          |
| R1.5g  | Enable client side sentiment analysis | Moderator     | Event - sentimentSenderAnalysis                                                       |                                                                                                                                                                                                                                                                                               |
| R1.5h  | Enable server side sentiment analysis | Moderator     | Event - sentimentModeratorAnalysis                                                    |                                                                                                                                                                                                                                                                                               |
| R1.6   | Moderator interaction                 | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.6a  | Moderate chat and questions           | Moderator     | Event - chatEnabled; Event - questionsEnabled; Message                                | An event can be configured for chat and question usage. If either is activated the Message entity represents a single chat or question which allows further requirements to both.                                                                                                             |
| R1.6b  | Moderator supports the Presenter      | Moderator     | Message - state; Message - presenterHint                                              | A message can be set to a specific state. The state "forwarded" represents the support of a Presenter by its Moderator. "forwarded" messages are the workbasket for the Presenter. The presenterHint can hold information for the Presenter.                                                  |
| R1.6c  | Answer attendee inputs                | Moderator     | Message - replyTo                                                                     | The replyTo relation of a message to another message allows the chaining of messages used for answering a specific message.                                                                                                                                                                   |
| R1.6d  | Moderator chats and questions         | Moderator     | Message - senderName                                                                  | A message from a Moderator does not differ much from a message of an Attendee. Only difference is that the senderName is "Moderator" in such cases.                                                                                                                                           |
| R1.6e  | Auto accept proper attendee inputs    | -             | Event - sentimentModeratorAnalysis; Event - sentimentModeratorAutoAccept              | Messages will be set to state "accepted" if an event has an active sentiment moderator analysis and an active auto approve flag when the sentiment analysis is positive.                                                                                                                      |
| R1.6f  | Auto reject improper attendee inputs  | -             | Event - sentimentModeratorAnalysis; Event - sentimentModeratorAutoReject              | Messages will be set to state "rejected" if an event has an active sentiment moderator analysis and an active auto reject flag when the sentiment analysis is negative.                                                                                                                       |
| R1.6g  | Export attendee inputs                | Manager       | Message                                                                               | Messages of an event can be exported to Excel including timestamp, state, likes, text and other attributes.                                                                                                                                                                                   |
| R1.7   | Browser access                        | Everyone      | -                                                                                     | Browser access is available by using state of the art web applications. We will focus currently on vue3.                                                                                                                                                                                      |
| R1.8   | Individual event access               | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.8a  | Support individual event access       | Attendee      | Event - eventId                                                                       | The eventId of an Event is unique and used as url parameter "event" for a direct access to the event. Result of this direct access is the user validation by asking for the user email that wants to access the event.                                                                        |
| R1.8b  | Support personalized event access     | Attendee      | Event - eventId; Event - accessList; User - email                                     | A second url parameter "user" enables the direct event access of a specific user to a event as long as he is part of the accessList of the event. Result of this direct access is the user validation by asking for the access token validation.                                              |
| R1.8c  | Support automatic event access        | Attendee      | Event - eventId; Event - accessList; User - email; AuthorizationToken                 | A direct access to an event without any user validation is possible by using the url parameter "token" and providing the proper access token together with the "event" and "user" url parameter. Result should be the direct access to the event for the user without any further validation. |
| R1.8d  | Support direct event track access     | Attendee      | Event - eventId; Event - channels; Track - trackId                                    | In combination with R1.8a, R1.8b and R1.8c an access to a specific track of the event is possible be using the additional url parameter "track" that should match a defined trackId of a Channel.                                                                                             |
| R1.9   | Compliance and user consent           | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.9a  | Data processing in EU                 | -             | -                                                                                     | Data center is in Nürnberg, Germany; Software is provided without any third party interaction.                                                                                                                                                                                                |
| R1.9b  | User consent                          | Attendee      | -                                                                                     | User consent is not implemented due to privacy by design. We proceed personal data only for the attendance and interaction reasons only as long as an event lasts. On event finish all data is anonymized as documented in the operations manual.                                             |
| R1.9c  | Login message                         | Attendee      | Event - loginInfo                                                                     | When accessing an event a login information can be optionally provided. This relies on the way an event was accessed - see R1.8a, R1.8b, R1.8c. The text will support HTML syntax.                                                                                                            |
| R1.9d  | Interaction message                   | Attendee      | Event - interactionInfo                                                               | This optional information will be displayed in the interaction part of the attendee interface. The text will support HTML syntax.                                                                                                                                                             |
| R1.10  | Streaming provider independent        | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.10a | Configure multiple streaming provider | Moderator     | Event - channels                                                                      | The channels relation enables multiple configurations for an event.                                                                                                                                                                                                                           |
| R1.10b | Switching streaming provider          | Moderator     | Event - activeProvider                                                                | The activeProvider allows a Manager to configure the current active channel group in order to switch to another streaming provider.                                                                                                                                                           |
| R1.11  | Event creation                        | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.11a | Import Ventari Excel                  | Manager       | Event - accessList; AuthorizationToken                                                | The excel import is handled by the Manager interface. When processing the excel file the accessList of the event gets set and AuthorizationTokens will be generated in state "pre-issued". Multiple import should be possible and already invited users should not be added multiple times. Already created Tokens are still returned.|
| R1.11b | Return URL to Ventari                 | Manager       |                                                                                       | The generated URL from R1.11a will be returned to the excel import as another excel file in response.                                                                                                                                                                          |
| R1.12  | Event statistics                      | -             | -                                                                                     | Requirement is detailed with its sub requirement                                                                                                                                                                                                                                              |
| R1.12a | General event statistics              | Manager       | Event - statistics; EventStatistic - timestamp; EventStatistic - numberOfConnections  | The event statistics are cumulated values at the moment of creation of the EventStatistic object. The number of logged-in-users at a given time is stored in the numberOfConnections attribute.                                                                                               |
| R1.12b | Debugging statistics                  | Manager       | EventStatistics                                                                       | This requirement is realized by all numberOf* attributes in the EventStatistic entity.                                                                                                                                                                                                        |
| R1.12c | Track statistics                      | Manager       | TrackStatistic                                                                        | Beside the EventStatistic which provides overall cumulated values, the TackStatistic allows a cumulated view of a specific Track of the event.                                                                                                                                                |
| R1.13  | Changes to event configurations       | Manager       | -                                                                                     | Changes to an event are propagated using web sockets to connected clients. The software must react on changes properly. |
| R1.14  | Presenter dashboard                   | Presenter     |                                                                                       | not designed yet                                                                                                                                                                                                                                                                              |
| R2.2   | Feedback                              | Attendee      |                                                                                       | not designed yet                                                                                                                                                                                                                                                                              |
| R2.3   | Cost reduction                        | -             | -                                                                                     | This is supported by building this solution instead of using Ventari and by operating the solution at Hetzner in Nürnberg instead of Azure or AWS.                                                                                                                                            |
| R2.4   | Mobile device support                 | Attendee      |                                                                                       | not designed yet                                                                                                                                                                                                                                                                              |
